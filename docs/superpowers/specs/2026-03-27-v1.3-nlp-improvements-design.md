**Date:** 2026-03-27
**Status:** Draft

## Goal

Close the NLP gap that causes 22% of natural-language prompts to miss template injection. Current engine: 10% injection rate on conversational prompts. Target: 80%+ with zero false positive increase.

## F1: Conversational Preprocessor

### Architecture

- **Location:** Inline `def preprocess_prompt` in `scripts/match.jq`, inserted between helper functions and Phase 0 (leave-it-alone)
- **Placement:** Phase 0.5 — runs AFTER leave-it-alone on prompts that survive detection
- **Output:** Metadata sidecar, NOT prompt replacement. Returns structured object consumed by Phase 1 and Phase 2 as additive signal
- **Observability:** Output exposed as `preprocessed` field in match.jq result JSON for testing

### Output Contract

```jq
def preprocess_prompt:
  {
    inferred_action: "debug" | "explain" | "optimize" | ... | null,
    inferred_object: "API" | "database" | ... | null,
    cleaned_terms: ["debug", "API"],        # additional terms for Phase 1 index lookup
    pattern_matched: "something_wrong" | ... | null
  };
```

### 9 Patterns (Priority Order)

Patterns are evaluated first-match-wins. Priority ordering prevents misclassification (e.g., P1 catches "what is wrong" as debug before P5 could classify it as explain).

**P1: "something is wrong" / diagnostic complaint → debug**

```
Regex (jq ONIG):
^(something (is |seems |looks |appears )?(wrong|broken|off|weird|messed up|not right)|
  there('s| is| are| seems to be) (a |an |some )?(problem|issue|bug|error)|
  (it|this|that) (is |keeps |won't stop )(not )?(working|running|loading|compiling|building|failing|crashing|breaking|erroring|responding|connecting)|
  what (is|went) wrong)

Extraction:
  inferred_action: "debug"
  inferred_object: scan remainder for "with|in|on" + known object

Examples:
  "something is wrong with the API" → debug, API
  "it keeps crashing" → debug, null
  "what went wrong" → debug, null
```

**P2: "why is X failing/crashing" → debug (with problem indicator)**

```
Two-part check:
  1. ^why (is|are|does|do|did|isn't|aren't|doesn't|don't|didn't|won't|can't|couldn't)
  2. Prompt contains problem indicator: (not |n't |wrong|broken|fail|crash|miss|slow|error|bug|hang|freeze|timeout|exception|undefined|null|stuck)

Both must match. "why is the sky blue" fails part 2 (no problem indicator).

Extraction:
  inferred_action: "debug"
  inferred_object: words between auxiliary and problem indicator, stripped of articles

Examples:
  "why is the build failing" → debug, "build"
  "why is the sky blue" → NO MATCH (no problem indicator)
  "why isn't the database connecting" → debug, "database"
```

**P3: "make it X-er" / "make X more Y" → adjective-to-action map**

```
Regex 3a: ^make (it|this|that) (.+)$
Regex 3b: ^make (.+) more (.+)$

Extraction:
  Look up captured adjective/verb in intents.adjective_actions or intents.verb_fix_map
  If found: inferred_action = mapped action
  If not found: no match, fall through

  For 3a: inferred_object = null (no object from "it")
  For 3b: inferred_object = group 1, stripped of articles

Examples:
  "make it faster" → optimize, null (via adjective_actions["faster"])
  "make it work" → fix, null (via verb_fix_map["work"])
  "make the API more robust" → fix, "API" (via adjective_actions["robust"])
  "make it purple" → NO MATCH ("purple" not in any map)
```

**P4: "how does X work" → explain**

```
Regex: ^how (does|do|did|would|should|could) (.+) (work|function|operate|run|execute|behave)

Terminal verb set is closed — "look" does NOT match, preventing
"how does this look" from being misclassified as explain.

Extraction:
  inferred_action: "explain"
  inferred_object: group 2, stripped of articles (the/a/an/this/that/my)

Examples:
  "how does the auth flow work" → explain, "auth flow"
  "how does this look" → NO MATCH ("look" not in terminal set)
  "how does this function run" → explain, "function"
```

**P5: "what is/are X" → explain (with first-person guard)**

```
Regex: ^what (is|are|was|were) (.+)

GUARD: Before defaulting to explain, scan remainder for:
  1. A recognized action verb (write/create/build/review/debug/fix/test/refactor/optimize/design/document)
  2. AND first-person framing (I/my/we/our/you) or imperative ("should", "need to", "want to")

If guard matches: use the detected action verb instead of "explain"
If guard does not match: inferred_action = "explain"

This prevents "what are best practices for error handling" → explain (correct, knowledge question)
While allowing "what are the tests I should write" → test (correct, task request)

Extraction:
  inferred_action: "explain" or detected action verb
  inferred_object: group 2, stripped of articles

Examples:
  "what is a closure" → explain, "closure"
  "what are the tests I should write" → test, "tests" (guard: "I should" + "write")
  "what are best practices for caching" → explain, "best practices for caching"
  "what is wrong" → ALREADY CAUGHT by P1
```

**P6: "[action] the Y I made/wrote" → filler stripping**

```
Precondition: first word must be a recognized canonical action or action synonym
(validated against intents.actions + intents.action_synonyms keys)

Filler regex (applied to remainder after action word):
  \s+(?:that\s+)?(?:i|we|you)\s+(?:have\s+)?(?:made|wrote|changed|created|built|modified|updated|added|fixed|pushed|committed|submitted|drafted)

Strips the filler clause, keeps everything before and after it.

Extraction:
  inferred_action: first word (already validated as canonical/synonym)
  cleaned_terms: prompt with "I made/wrote/..." clause removed

Examples:
  "review the changes I made to checkout" → review, cleaned="review the changes to checkout"
  "check the code I wrote yesterday" → review (via synonym), cleaned="check the code yesterday"
  "discuss the plan I made" → NO MATCH ("discuss" not a recognized action)
```

**P7: "can you X" → prefix strip**

```
Regex: ^can you (please )?(help (me|us )?(to |with )?)?(.+)$

Minimum: remainder (group 5) must have >= 2 non-stopword tokens
"can you help" → remainder too short → NO MATCH

Extraction:
  inferred_action: null (pass-through, let existing scoring handle)
  inferred_object: null
  cleaned_terms: tokenized remainder appended to index lookup

Examples:
  "can you review my code" → cleaned_terms=["review","my","code"] (redundant but harmless)
  "can you help me debug this" → cleaned_terms=["debug","this"]
  "can you help" → NO MATCH (too short)
```

**P8: "I need to X" → prefix strip**

```
Regex: ^i need to (.+)$

Minimum: remainder must have >= 2 non-stopword tokens

Extraction:
  inferred_action: null
  cleaned_terms: tokenized remainder

Examples:
  "I need to refactor the database schema" → cleaned_terms=["refactor","the","database","schema"]
  "I need to go" → cleaned_terms=["go"] → 1 token → NO MATCH
```

**P9: "help me X" → prefix strip**

```
Regex: ^help (me|us )?(to )?(with )?(.+)$

Minimum: remainder (group 4) must have >= 2 non-stopword tokens

Extraction:
  inferred_action: null
  cleaned_terms: tokenized remainder

Examples:
  "help me refactor this function" → cleaned_terms=["refactor","this","function"]
  "help" → NO MATCH (no remainder)
```

### Scoring Integration

In `score_candidate`, add inferred bonuses using max() — never additive:

```jq
# Inferred action bonus (per-template: only fires if template.action matches)
(if ($pp.inferred_action != null) and ($pp.inferred_action == ($tmpl.action | ascii_downcase))
 then 0.30
 else 0 end) as $inferred_action_score |

# Inferred object bonus (per-template)
(if ($pp.inferred_object != null) and
    (($pp.inferred_object | ascii_downcase) as $io |
     ($tmpl.object | ascii_downcase) as $to |
     ($io == $to) or (is_object_synonym($io; $to)))
 then 0.20
 else 0 end) as $inferred_object_score |

# Final scores: max of literal vs inferred
([$action_score, $inferred_action_score] | max) as $effective_action |
([$object_score, $inferred_object_score] | max) as $effective_object |
```

**Scoring budget (v1.3):**

| Component | Value | Source |
|-----------|-------|--------|
| Action (literal or inferred) | max(0.45, 0.30) | Literal wins when present |
| Object (literal or inferred) | max(0.35, 0.20) | Literal wins when present |
| Keyword overlap | +0.02/hit, cap 0.08 | Unchanged |
| Intent signal | +0.10 cap | Unchanged |
| Negative signal | -0.30 each (uncapped) | Unchanged |
| Complexity penalty | -0.15 (or -0.30 for <3 words, see F9) | Enhanced |
| Multi-turn suppression | -0.40 | Unchanged |
| **Max theoretical** | **0.98** | **Unchanged from v1.2** |
| **Pure inference ceiling** | **0.68** | **Deliberately below inject (0.70)** |

The preprocessor can only trigger injection when combined with at least one literal word match. This is the correct safety model.

### Phase 1 Enhancement

`keyword_candidates` modified to include `cleaned_terms` in index lookup:

```jq
def keyword_candidates_enhanced($pp):
  prompt_words as $words |
  prompt_bigrams as $bigrams |
  ($pp.cleaned_terms // []) as $extra |
  .inverted_index as $idx |
  ([$words[] | . as $w | $idx[$w] // [] | .[]] +
   [$bigrams[] | . as $b | $idx[$b] // [] | .[]] +
   [$extra[] | . as $e | $idx[$e] // [] | .[]]) |
   group_by(.) | map({ id: .[0], hits: length }) | sort_by(-.hits);
```

## F2: Shell_command "make" Regex Fix

**Bug:** Current regex `^(git|npm|...|make|...)\\s` suppresses all prompts starting with "make " including natural language like "make it faster", "make a new component", "make the code cleaner".

**Fix:** Replace the blanket "make" entry with an allowlist of Makefile targets:

```jq
# Before (v1.2):
^(git|npm|yarn|pnpm|pip|cargo|make|docker|kubectl|cd|ls|cat|mkdir|mv|cp|rm|chmod|curl|wget|ssh|scp)\\s

# After (v1.3):
^(git|npm|yarn|pnpm|pip|cargo|docker|kubectl|cd|ls|cat|mkdir|mv|cp|rm|chmod|curl|wget|ssh|scp)\\s|^make\\s+(test|install|build|clean|all|deploy|lint|dist|run|check|help|format)([\\s]|$)
```

"make" only triggers shell_command when followed by a known Makefile target. "make it faster", "make a component", "make the code cleaner" all pass through.

## F3: Case-Insensitive Index + Scoring

**Bug (double):**
1. `build-index.sh` stores uppercase index keys ("API", "PR", "TypeScript"). Lookups use lowercase. 310 dead keys.
2. `score_candidate` compares lowercased prompt words against original-case template `.action`/`.object`. "api" never matches "API".

**Fix 1 — build-index.sh (lines 224-244):**

Lowercase all keys when building the inverted index:
- `$trigger | ascii_downcase` before indexing
- `$tmpl.action | ascii_downcase` and `$tmpl.object | ascii_downcase` when indexing by action/object
- Add individual words from triggers after lowercasing

**Fix 2 — match.jq score_candidate:**

Lowercase template fields at comparison time:
```jq
($tmpl.action | ascii_downcase) as $action |
($tmpl.object | ascii_downcase) as $object |
# Use $action and $object for all comparisons instead of $tmpl.action/$tmpl.object
```

**Validation:** After rebuilding index, assert: `jq '.inverted_index | keys | map(select(test("[A-Z]"))) | length' data/index.json` = 0

**Risk:** Zero. Strictly additive — fixes silent misses, no regressions. All prompt words already lowercased.

## F5: Morphological Matching

**Approach:** Curated `morphological_map` in intents.json v2, NOT open suffix stripping.

The stress tester found that open stemming creates false positives ("classify" ↔ "class", "running" ↔ "rune") and is impractical in pure jq. A curated map is bounded, controllable, and expandable based on telemetry.

**Integration points:**

1. **Phase 1 (keyword extraction):** Before index lookup, check if word is in morphological_map. If yes, also look up the normalized form:
```jq
# For each prompt word, also look up its morphological root
($morph_map[$w] // null) as $root |
if $root != null then
  (($idx[$w] // []) + ($idx[$root] // []))
else
  ($idx[$w] // [])
end
```

2. **Phase 2 (synonym matching):** In `is_action_synonym` and `is_object_synonym`, normalize both the prompt word and the synonym via the map before comparison:
```jq
(($morph_map[$word] // $word)) as $norm_word |
(($morph_map[$syn] // $syn)) as $norm_syn |
$norm_word == $norm_syn
```

**Data source:** `intents.morphological_map`, loaded via existing `--slurpfile intents`.

## F6: Code Snippet Detector

**Problem:** `"const review = (code) => test(code)"` scores 0.55 for code review — false positive. JavaScript identifiers named "review", "code", "test" trigger action/object matching.

**Solution:** Add a `code_snippet` heuristic to Phase 0 leave-it-alone detector.

**Heuristic:**
```jq
# Code snippet detection
# Count programming punctuation characters: ; { } ( ) [ ] = < > => ->
# If density > 1 per 8 words AND no question mark, score as code
(($p | [scan("[;{}()\\[\\]=<>]")] | length) as $punct_count |
 if $punct_count > 0 and (prompt_word_count > 0) and
    ($punct_count / prompt_word_count > 0.125) and
    ($p | test("\\?") | not)
 then { score: 0.40, reason: "code_snippet" }
 else null end),
```

Score 0.40 alone doesn't suppress (threshold 0.60), but it compounds with other signals. Code snippet + numbered list (0.15) = 0.40 + 0.15*0.3 = 0.445 (still passes). Code snippet + long_structured (0.45) = 0.45 + 0.40*0.3 = 0.57 (still passes but close). Code snippet + shell_command (0.60) = 0.60 (suppresses — correct, it's code).

The 0.40 score is conservative by design — code with questions ("why does this throw?") should still be matchable.

## F7: P2 First-Person Guard

**Problem:** Without a guard, P5 ("what is/are X") always defaults to "explain", misclassifying task requests like "what are the tests I should write" and over-classifying knowledge questions.

**Rule:** Only override "explain" with a task action when the remainder contains:
1. A recognized action verb (from intents.actions + action_synonyms keys)
2. AND first-person/imperative framing: `(i|my|we|our|you)` or `(should|need to|want to|have to|going to|gonna|gotta)`

**Examples:**
| Prompt | Has action verb? | Has first-person? | Result |
|--------|-----------------|-------------------|--------|
| "what are the tests I should write" | write (→create/test) | "I should" | → test |
| "what is the code I need to review" | review | "I need to" | → review |
| "what are best practices for caching" | No | No | → explain |
| "what is a closure" | No | No | → explain |
| "what are the API endpoints" | No | No | → explain |

## F8: Synonym Prefix Overlap Ratio

**Problem:** Current bidirectional prefix matching (`startswith`) is too loose. As synonym lists grow, "class" could match "classify", "test" could match "testing" at only 57% overlap.

**Fix:** Add minimum overlap ratio requirement:

```jq
# In is_action_synonym and is_object_synonym, replace:
(($word | startswith($syn)) or ($syn | startswith($word)))

# With:
(($word | startswith($syn)) or ($syn | startswith($word))) and
(([($word | length), ($syn | length)] | min) / ([($word | length), ($syn | length)] | max) >= 0.75)
```

Requires the shorter string to be at least 75% of the longer string's length. This blocks:
- "class" (5) / "classify" (8) = 62% → blocked
- "test" (4) / "testing" (7) = 57% → blocked

While allowing:
- "optimize" (8) / "optimizing" (10) = 80% → allowed
- "review" (6) / "reviewer" (8) = 75% → allowed

Note: morphological_map (F5) handles the cases that overlap ratio blocks. "testing" → "test" via map, then exact-matches. This is the intended design — curated map for known inflections, overlap ratio as safety net for unknown words.

## F9: Single-Word Prompt Penalty

**Problem:** Single-word prompts like "test" score 0.45 (defer) — too aggressive for a bare word that's likely a shell command or continuation.

**Fix:** Increase complexity penalty for very short prompts:

```jq
# Before (v1.2):
(if (prompt_word_count < 6) and ($tmpl.min_confidence > 0.7) then -0.15 else 0 end)

# After (v1.3):
(if (prompt_word_count < 3) then -0.30
 elif (prompt_word_count < 6) and ($tmpl.min_confidence > 0.7) then -0.15
 else 0 end)
```

Effect: "test" alone = 0.45 - 0.30 = 0.15 (skip). "test this" = 0.45 - 0.30 = 0.15 (skip). "test this function" = 0.45 - 0.15 = 0.30 (skip) or no penalty if min_confidence ≤ 0.7. Three words is the minimum for meaningful intent.

## F12: Compound Intent Telemetry

**Problem:** "fix the bug and add tests" picks one winner, ignoring the second intent.

**v1.3 scope:** Don't solve multi-intent composition. Instead, detect and log compound prompts for v2.0 prioritization.

**Detection:** After scoring, if the top-2 candidates both score >0.50 AND have different actions:
```jq
if ($candidates[0].confidence > 0.50) and ($candidates[1].confidence > 0.50) and
   ($candidates[0].action != $candidates[1].action)
then { compound_intent_detected: true, intents: [$candidates[0].action, $candidates[1].action] }
else { compound_intent_detected: false }
end
```

Write to telemetry log alongside normal match result. No behavioral change — just data collection.
