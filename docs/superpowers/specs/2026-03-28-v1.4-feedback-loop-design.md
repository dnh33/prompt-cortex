# v1.4 — Feedback Loop & Scoring Refinement

> **Status**: Approved scope, pending implementation plan
> **Date**: 2026-03-28
> **Author**: Daniel Hjermitslev + Rune (Claude Opus 4.6)
> **Method**: 5-agent AI council (Scoring Architect, NLP Engineer, UX Expert, Systems Pragmatist, Feedback Loop Designer)

---

## Executive Summary

v1.4 closes the scoring gap that v1.3's preprocessor exposed and adds the minimum viable feedback loop. Six features, one session.

The core insight from the council: v1.3's preprocessor correctly detects intent for conversational prompts but the scoring formula can't convert detection into injection. The fix is NOT lowering thresholds globally — it's per-action thresholds (debug is low-risk, design is high-risk) combined with fixing the P6 silent swallow bug that blocks direct imperatives.

---

## Architecture Context

**Current scoring formula (v1.3):**
```
confidence = max(action_score, inferred_action) + max(object_score, inferred_object)
           + keyword + signal + penalties
```

**Current thresholds:** inject ≥ 0.70, defer ≥ 0.40

**The gap:** Prompts like "make it faster" (pp=optimize, conf=0.35) and "document the endpoints" (conf=0, P6 swallows it) correctly detect intent but can't inject.

**v1.4 approach:** Don't lower the floor. Raise the ceiling for low-risk actions and fix the patterns that silently fail.

## Feature Specifications

### F1: Per-Action Thresholds

**Problem:** All actions share the 0.70 inject threshold, but actions have different risk profiles. Injecting a debug template on a marginal match is almost always helpful. Injecting a design template on a marginal match can misdirect work.

**Solution:** Replace the single `effective_threshold` with a per-action threshold lookup.

**Thresholds (from Scoring Architect analysis):**

| Action | Threshold | Rationale |
|--------|-----------|-----------|
| debug | 0.62 | Debug help is contextually appropriate in almost all cases |
| fix | 0.63 | Same rationale as debug — fixing is low-risk |
| explain | 0.64 | Explain templates are lightweight, rarely harmful |
| refactor | 0.65 | Code quality guidance rarely hurts |
| optimize | 0.65 | Performance guidance rarely hurts |
| document | 0.66 | Documentation templates are informational |
| test | 0.67 | Test templates suggest structure, moderate risk |
| review | 0.70 | Review templates are substantive — keep current threshold |
| create | 0.70 | Creation templates direct significant work |
| design | 0.73 | Architecture injection can misdirect — raise above current |

**Implementation in match.jq:**

```jq
def action_thresholds:
  { "debug": 0.62, "fix": 0.63, "explain": 0.64,
    "refactor": 0.65, "optimize": 0.65, "document": 0.66,
    "test": 0.67, "review": 0.70, "create": 0.70, "design": 0.73 };
```

In the main pipeline, replace the threshold check:
```jq
# Current:
if $best.confidence >= ([effective_threshold, $min_conf] | max) then "inject"

# New:
($best.action // "review") as $best_action |
(action_thresholds[$best_action] // 0.70) as $base_threshold |
(($min_confidence_adjust // "0") | tonumber | [., 0.10] | min | [., -0.10] | max) as $adj |
($base_threshold + $adj) as $action_threshold |
if $best.confidence >= ([$action_threshold, $min_conf] | max) then "inject"
```

The preset `min_confidence_adjust` still applies on top, so presets remain effective.

**Also fold in (from NLP Engineer):** Add `is_domain_object_synonym` check to inferred_object scoring. Currently line ~496 checks `is_object_synonym` but NOT `is_domain_object_synonym`. One-line addition:

```jq
# Current:
($io == $object) or (is_object_synonym($io; $object))

# New:
($io == $object) or (is_object_synonym($io; $object)) or (is_domain_object_synonym($io; $object))
```

**Tests:** ~8 new
- "something is wrong with the API" → inject (debug threshold 0.62, conf should reach with domain object boost)
- "it keeps crashing" → test actual behavior (may still defer, document either way)
- "make it faster" → test actual behavior
- "design a new architecture" → verify higher threshold (0.73) prevents marginal inject
- Verify all existing tests still pass (threshold changes could regress)
- Test with preset adjust: `min_confidence_adjust: "-0.05"` → thresholds shift down

**Files:** `scripts/match.jq` (scoring + pipeline)

### F2: P6 Fix — Direct Imperative Extraction + Filler Tail Preservation

**Problem (P6 silent swallow):** When a prompt starts with a recognized action word but has no filler suffix ("I made/wrote/etc"), P6 returns `{inferred_action: null}` AND blocks P7-P9 from firing. Direct imperatives like "document the endpoints" get swallowed.

**Problem (filler tail loss):** P6's filler regex uses `.*` which strips everything after "I made", including useful context. "review the changes I made to checkout" loses "to checkout".

**Solution: Two fixes in one task.**

**Fix A — Direct imperative extraction:**

When P6 detects a first-word action but finds no filler, instead of returning null, extract the action and remaining words as object:

```jq
# In P6's else branch (when $cleaned == $p, meaning no filler found):
# Current: returns null
# New: extract direct imperative
($p | split(" ") | .[1:] | map(select(length > 0)) |
 map(select(. as $w | ["the","a","an","this","that","my","our","your","its","please","just","now","quickly","for","to","from"] | index($w) == null))) as $obj_words |
if ($obj_words | length) >= 1 then
  { inferred_action: $canonical_action,
    inferred_object: ($obj_words | join(" ")),
    cleaned_terms: ([$canonical_action] + $obj_words),
    pattern_matched: "direct_imperative" }
else
  { inferred_action: $canonical_action, inferred_object: null,
    cleaned_terms: [$canonical_action], pattern_matched: "direct_imperative" }
end
```

**Fix B — Filler tail preservation:**

Replace the greedy `.*` in the filler strip regex with a bounded strip:

```jq
# Current:
gsub("\\s+(?:that )?(?:i|we|you) (?:have )?(?:made|wrote|changed|created|built|modified|updated|added|fixed|pushed|committed|submitted|drafted).*"; "")

# New (stop at verb+optional adverb, preserve prepositional tail):
gsub("\\s+(?:that )?(?:i|we|you) (?:have )?(?:made|wrote|changed|created|built|modified|updated|added|fixed|pushed|committed|submitted|drafted)(?:\\s+(?:recently|just|already|so far|earlier|yesterday|today))?(?=\\s|$)"; "")
```

ONIG supports lookahead `(?=...)`. This preserves "to checkout", "for the cart", etc.

**Tests:** ~8 new
- "document the endpoints" → inferred_action: "document", pattern: "direct_imperative"
- "test the payment flow" → inferred_action: "test", pattern: "direct_imperative"
- "create a new config file" → inferred_action: "create", pattern: "direct_imperative"
- "review the changes I made to checkout" → cleaned preserves "to checkout"
- "debug the function I wrote for the payment API" → preserves "for the payment API"
- "fix" (single word, no object) → still returns inferred_action, no object
- Verify P3a/P3b still fire before P6 for "make" prompts
- Verify P7-P9 are unreachable only when P6 fires (not swallowed)

**Files:** `scripts/match.jq` (preprocess_prompt function, P6 block)

### F3: P11 — Subject-Predicate Diagnostic Pattern

**Problem:** P1 catches "something is wrong" and "it keeps crashing" but misses "the payment form is broken", "login is failing", "our CI pipeline is broken" — real diagnostic complaints with explicit subjects.

**Solution:** Add P11 after P9, before the else fallthrough:

```jq
# P11: "X is/keeps broken/failing" → debug (subject-predicate diagnostic)
elif ($p | test("(is|are|keeps|keep|started|has been|have been) (broken|failing|crashing|hanging|dropping|erroring|throwing|timing out|not working|not responding|not loading|not starting|not compiling|acting up|misbehaving)"))
then
  ($p | capture("^(?<subj>.+?)\\s+(?:is|are|keeps|keep|started|has been|have been)\\s+(?:broken|failing|crashing|hanging|dropping|erroring|throwing|timing out|not working|not responding|not loading|not starting|not compiling|acting up|misbehaving)") // {}) as $cap |
  ($cap.subj // null | if . then strip_articles else null end) as $subject |
  { inferred_action: "debug", inferred_object: $subject,
    cleaned_terms: (["debug"] + (if $subject then [$subject | split(" ")[]] else [] end)),
    pattern_matched: "subject_broken" }
```

**Also fold in:** Expand P2's problem indicator list with missing terms: add `breaking`, `broke`, `stalling` to the regex.

**False positive analysis (from NLP Engineer):**
- "my architecture is clean" → "clean" NOT in negative list → safe
- "the design is complete" → "complete" NOT in list → safe
- "my test coverage is low" → "low" NOT in list → safe
- "the build is taking forever" → "taking forever" NOT in list (no "taking") → safe, but could consider adding

**Tests:** ~5 new
- "the payment form is broken" → inferred_action: "debug", inferred_object: "payment form"
- "login is failing for some users" → inferred_action: "debug", inferred_object: "login"
- "our CI pipeline is broken" → inferred_action: "debug"
- "the database connection keeps dropping" → inferred_action: "debug"
- Negative: "the design is complete" → no pattern matched

**Files:** `scripts/match.jq` (preprocess_prompt, add P11 + expand P2)

### F4: Session-Immediate Negative Feedback Penalty

**Problem:** `/cortex:feedback bad` logs to usage.jsonl but nothing changes. The same template can re-inject immediately. Users feel unheard.

**Solution:** When `/cortex:feedback bad` fires, add the template ID to session state's `penalizedTemplates` array. In `score_candidate`, apply -0.25 penalty for penalized templates. Resets on new session.

**In cortex-match (bash):** When writing feedback to usage.jsonl, also update state:

```bash
# After writing feedback to usage.jsonl:
if [[ "$feedback_rating" == "bad" ]]; then
  # Add to penalized list in state
  state=$(jq --arg tmpl "$last_template_id" \
    '.penalizedTemplates = ((.penalizedTemplates // []) + [$tmpl] | unique)' \
    "$state_file")
fi
```

**In match.jq:** After `$suppression_penalty`, before inferred scoring:

```jq
# --- Session feedback penalty (v1.4) ---
(parsed_state as $s |
 if $s == null then 0
 else
   if ($s.penalizedTemplates // []) | any(. == $tmpl.id) then -0.25
   else 0 end
 end) as $feedback_penalty |
```

Add to confidence total and breakdown.

**UX (from UX Expert):** The `/cortex:feedback bad` command response should acknowledge:

```
Noted. [template-id] penalized for this session.
It will need stronger signal to match again.
```

**Tests:** ~4 new
- Template with feedback penalty scores -0.25 lower
- Penalized template that would normally inject → defers or skips
- Non-penalized templates unaffected
- State without penalizedTemplates field → no crash (null safety)

**Files:** `scripts/match.jq` (score_candidate), `hooks/cortex-match` (state write), `commands/feedback.md` (response text)

### F5: `/cx off` Persistence Fix

**Problem:** `cortex-match` line 302 hardcodes `cortex_disabled: false` in every state write. `/cx off` only works until the next prompt.

**Solution:** Read `cortex_disabled` from `.cortex/config.json` (where `min_tier` and `transparent_mode` already live) instead of hardcoding in state.

**In cortex-match:**

```bash
# Current (line ~302):
cortex_disabled: false

# New: read from config
cortex_disabled_val=$(jq -r '.cortex_disabled // false' "$CONFIG_FILE" 2>/dev/null || echo "false")

# In state JSON:
cortex_disabled: $cortex_disabled_val
```

**The /cx off command** should write to config.json, not state:

```bash
# In the cx off handler:
jq '.cortex_disabled = true' "$CONFIG_FILE" > "${CONFIG_FILE}.tmp" && mv "${CONFIG_FILE}.tmp" "$CONFIG_FILE"
```

**Tests:** ~3 new
- `/cx off` persists across hook invocations (simulate with state reload)
- `/cx on` restores normal behavior
- Session restart respects config setting

**Files:** `hooks/cortex-match` (state construction + cx handler)

### F6: Defer Visibility in Transparent Mode

**Problem:** When transparent mode is ON and cortex defers (0.40-0.69), the user sees nothing. They don't know a partial match happened. The "forgot about it" user misses that cortex tried to help.

**Solution:** When transparent mode is enabled and action is "defer", append a one-line footer to the hook output:

```
[cortex: deferred — partial match on {template_name} (confidence: {conf}). /cortex:show for full template]
```

**In cortex-match (bash):** After the defer path, check transparent mode:

```bash
if [[ "$action" == "defer" ]] && [[ "$transparent_mode" == "true" ]]; then
  # Add defer hint to outputContent (or additionalContext footer)
  defer_msg="[cortex: deferred — partial match on ${best_id} (${confidence}). /cortex:show for full template]"
  # Append to additionalContext or outputContent
fi
```

**Design decision (from UX Expert):** Only show when transparent mode is ON. Default (transparent OFF) stays silent. This respects the "forgot about it" user while giving power users visibility.

**Tests:** ~3 new
- Defer with transparent mode ON → footer appears in output
- Defer with transparent mode OFF → no footer (current behavior)
- Inject with transparent mode ON → existing inject footer (no change)

**Files:** `hooks/cortex-match` (defer output path)

## What's NOT in v1.4 (Deferred to v2.0)

| Feature | Reason |
|---------|--------|
| Cross-session adaptive confidence | Requires Bayesian scoring, new file, SessionStart computation. Full design by Feedback Loop Designer ready for v2.0. |
| Multi-template composition | Fundamentally different algorithm (multi-winner scoring). Part of v2.0 Layer 2. |
| Template pack marketplace | Too much overhead. Community packs via git PRs instead. |
| Prompt history command | Low UX impact. `/cortex:stats` covers the diagnostic use case. 30-line bash script if needed. |
| Preset auto-detection | Useful but risky without escape hatches. Deferred until cross-session data validates the signals. |
| P12+ patterns | Ceiling of rule-based NLP reached. More patterns = more maintenance, diminishing returns. Invest in data (triggers, intent_signals, synonyms) not code. |
