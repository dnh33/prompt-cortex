> **Status**: Approved design, pending implementation plan

> **Author**: Daniel Hjermitslev + Rune (Claude Opus 4.6)
> **Repo**: dnh33/prompt-cortex

## Executive Summary

v1.2 makes prompt-cortex context-aware. The matching engine gains project intelligence — detected tech stack, CLAUDE.md conventions, user-configured presets, and team-shared templates all feed into template selection. The result: a TypeScript/Next.js project automatically gets TS-relevant templates boosted and Python templates suppressed, without the user configuring anything.

Two pillars:
1. **Smarter Matching** — wire the Phase 3 context_filter stub, add project affinity scoring, complexity assessment, domain synonyms, and reactive context detection via new hooks API
2. **Extensibility** — .cortex/project.json config, presets system, /cortex:suggest LLM-powered CLAUDE.md analysis (A+B hybrid: explicit command + cached results), template packs, team-shared templates

### Competitive Context

Research confirms this is an unoccupied gap. No Claude Code plugin does project-type-aware template selection. Cline has glob-based conditional rule activation (closest reference), claude-mem (40.9k stars) does session memory, prompt-improver (1.3k stars) clarifies vague prompts — but nobody selects from a template library based on detected stack. v1.2 fills this gap.

## Architecture Changes

### New Hook Events (from Hooks API updates)

v1.2 leverages new hooks API capabilities. **IMPORTANT**: CwdChanged, FileChanged, and CLAUDE_ENV_FILE must be verified against the actual Claude Code hooks documentation before implementation. If unavailable, the fallback is:
- **CwdChanged fallback**: cortex-match compares `$cwd` against last-known cwd in state file on every UserPromptSubmit. If different, re-run detection inline. Cheap — single string comparison.
- **FileChanged fallback**: Not critical. SessionStart detection covers initial load. Re-detection only matters for long sessions where manifests change mid-session (rare).
- **CLAUDE_ENV_FILE fallback**: Write detected context to `.cortex/session-context.json` instead. cortex-match reads this file directly (already reads state files from .cortex/).

| Hook/Feature | Purpose | Implementation | Fallback |
|---|---|---|---|
| `CwdChanged` | Re-detect project context when user cd's | New hook script: `cortex-cwd-changed` | cwd comparison in cortex-match |
| `FileChanged` | Re-detect when manifest files change | Triggered via `watchPaths` return from SessionStart | Skip — SessionStart is sufficient |
| `CLAUDE_ENV_FILE` | Persist detected context as env vars across session | Written by SessionStart, read by cortex-match | `.cortex/session-context.json` file |

**CLAUDE_ENV_FILE details** (if available): Claude Code sets the `CLAUDE_ENV_FILE` environment variable pointing to a writable file. Hooks write `KEY=VALUE` lines to this file. The values become available as environment variables in subsequent hook invocations within the same session. If `CLAUDE_ENV_FILE` is not set, fall back to writing `.cortex/session-context.json`.

### Updated Pipeline

```
SessionStart (fires once)
  ├─ Validate/rebuild index (existing)
  ├─ Detect tech stack (existing, enhanced)
  ├─ Load .cortex/project-context.json (NEW — cached LLM analysis)
  ├─ Load .cortex/project.json (NEW — manual config + presets)
  ├─ Write CLAUDE_ENV_FILE with detected context (NEW)
  └─ Return watchPaths for manifest files (NEW)

CwdChanged (fires on cd) — NEW
  ├─ Re-run tech stack detection
  ├─ Re-load project configs from new cwd
  └─ Update CLAUDE_ENV_FILE

FileChanged (fires on manifest edit) — NEW
  ├─ Re-run tech stack detection for changed file
  └─ Update CLAUDE_ENV_FILE

UserPromptSubmit (fires on every prompt)
  ├─ Phase 0: Leave-it-alone (existing)
  ├─ Phase 1: Inverted index lookup (existing)
  ├─ Phase 2: Score candidates (existing, + complexity scoring)
  ├─ Phase 3: Context filter (STUB → IMPLEMENTED)
  │   ├─ Filter by requires.language / requires.framework
  │   ├─ Boost by project_affinity (+0.05)
  │   ├─ Apply boost/suppress rules from project.json
  │   ├─ Apply preset boost/suppress
  │   ├─ Apply cached CLAUDE.md rules from project-context.json
  │   └─ Apply git branch context boost
  └─ Phase 4: Inject/Defer/Skip + output (existing)
```

### Context Passing

SessionStart and CwdChanged write context to `CLAUDE_ENV_FILE`:
```bash
CORTEX_LANG=typescript
CORTEX_FRAMEWORK=nextjs
CORTEX_TESTING=vitest
CORTEX_LINTER=biome
CORTEX_PKGMGR=bun
CORTEX_BRANCH_TYPE=feature
CORTEX_PRESET=greenfield
CORTEX_HAS_PROJECT_JSON=true
CORTEX_HAS_PROJECT_CONTEXT=true
```

cortex-match reads these env vars and passes them to match.jq as `$context`:
```bash
context_json=$(jq -n \
  --arg lang "${CORTEX_LANG:-unknown}" \
  --arg framework "${CORTEX_FRAMEWORK:-unknown}" \
  --arg testing "${CORTEX_TESTING:-}" \
  --arg branch_type "${CORTEX_BRANCH_TYPE:-}" \
  --arg preset "${CORTEX_PRESET:-}" \
  '{ lang: $lang, framework: $framework, testing: $testing,
     branch_type: $branch_type, preset: $preset }')

jq -f "$MATCH_JQ" \
  --argjson context "$context_json" \
  --argjson project_rules "$project_rules_json" \
  ... "$INDEX"
```

### Config Precedence

When multiple config sources conflict, the order is:

1. **Escape mechanisms** (--raw, /cx off) — always wins
2. **project.json** manual config — user's explicit rules
3. **project-context.json** cached LLM analysis — user-approved
4. **Active preset** — named configuration
5. **Auto-detected context** — regex-based tech stack
6. **Built-in defaults** — current v1.1 behavior

## Feature Specifications

### F1: Phase 3 Context Filter (match.jq)

**Current state**: No-op stub at match.jq:241-244
**Target**: Full context-aware filtering and scoring

**IMPORTANT**: `$context` and `$project_rules` are passed via `--argjson`. cortex-match MUST always pass both, even when no config exists — use `--argjson context '{}'` and `--argjson project_rules '{}'` as defaults. jq crashes on undeclared variables at compile time (same issue as `$min_tier` and `$intents`).

**IMPORTANT**: `context_filter` cannot be a `def` that references `.templates as $all_templates` — jq `as` bindings are lexically scoped and invisible inside `def`s. Implementation must either: (a) pass `$all_templates` as a parameter: `def context_filter($scored; $all_templates):`, or (b) inline the filter logic in the main pipeline instead of using a `def`. Option (a) is cleaner. The pseudocode below uses option (a).

```jq
# Phase 3: Context filter
# Called as: context_filter($scored; $all_templates) in main pipeline
def context_filter($scored; $all_templates):
  # Parse context and project rules (always passed, default to {})
  ($context // {}) as $ctx |
  ($project_rules // {}) as $rules |

  # 1. Language/framework filter — remove templates requiring wrong stack
  [ $scored[] |
    . as $entry |
    ($all_templates | map(select(.id == $entry.id)) | .[0]) as $tmpl |
    if ($tmpl.requires.language // []) | length > 0 then
      if ($tmpl.requires.language | map(ascii_downcase) | index($ctx.lang // "")) != null then $entry
      else empty end
    else $entry end
  ] |

  # 2. Framework filter
  [ .[] |
    . as $entry |
    ($all_templates | map(select(.id == $entry.id)) | .[0]) as $tmpl |
    if ($tmpl.requires.framework // []) | length > 0 then
      if ($tmpl.requires.framework | map(ascii_downcase) | index($ctx.framework // "")) != null then $entry
      else empty end
    else $entry end
  ] |

  # 3. Project affinity boost (+0.05 per matching affinity)
  [ .[] |
    . as $entry |
    ($all_templates | map(select(.id == $entry.id)) | .[0]) as $tmpl |
    (($tmpl.project_affinity // []) |
     map(select(. as $aff |
       ($ctx.lang == $aff) or ($ctx.framework == $aff) or
       ($ctx | tostring | test($aff; "i"))
     )) | length) as $affinity_hits |
    if $affinity_hits > 0 then
      .confidence = .confidence + ([$affinity_hits * 0.05, 0.10] | min)
    else . end
  ] |

  # 4. Boost/suppress rules from project.json + project-context.json
  [ .[] |
    . as $entry |
    ($all_templates | map(select(.id == $entry.id)) | .[0]) as $tmpl |
    # Boost matching categories, actions, or template IDs
    (if ($rules.boost // []) | any(. as $b | $tmpl.category == $b or $tmpl.action == $b or $tmpl.id == $b)
     then .confidence = .confidence + 0.05
     else . end) |
    # Suppress matching categories, actions, or template IDs
    (if ($rules.suppress // []) | any(. as $s | $tmpl.category == $s or $tmpl.action == $s or $tmpl.id == $s)
     then .confidence = .confidence - 0.15
     else . end) |
    # Disabled templates — remove entirely
    (if ($rules.disabled // []) | any(. == $entry.id or . == "*")
     then empty
     else . end)
  ] |

  # 5. Git branch context boost
  [ .[] |
    . as $entry |
    ($all_templates | map(select(.id == $entry.id)) | .[0]) as $tmpl |
    (if $ctx.branch_type == "feature" and ($tmpl.action == "design" or $tmpl.action == "create")
     then .confidence = .confidence + 0.03
     elif $ctx.branch_type == "fix" and ($tmpl.action == "debug" or $tmpl.action == "fix" or $tmpl.action == "test")
     then .confidence = .confidence + 0.03
     elif $ctx.branch_type == "refactor" and ($tmpl.action == "refactor" or $tmpl.action == "review")
     then .confidence = .confidence + 0.03
     else . end)
  ] |

  # Re-sort by confidence after adjustments
  sort_by(-.confidence);
```

**New jq args required**: `--argjson context` (default: `'{}'`), `--argjson project_rules` (default: `'{}'`). These MUST always be passed, even when empty — same pattern as `$min_tier` and `$intents`.

**Main pipeline call site change**: Replace `context_filter($scored)` with `context_filter($scored; $all_templates)` at match.jq:293.

### F2: Template Schema Upgrade

Add declared-but-unused fields to actual template files:

**Gold templates (78 files)** — full metadata:
```yaml
requires:
  language: [typescript, javascript]
  framework: []
  has_codebase: true
  min_complexity: medium
project_affinity:
  - web
  - api
```

**Silver templates (272 files)** — language/framework only:
```yaml
requires:
  language: []  # empty = any
project_affinity: []
```

**build-index.sh changes**: Parse `requires`, `project_affinity`, and `min_complexity` fields, include in index.json template objects.

**Bulk update strategy for 350 templates**:
- **Silver templates (272 files)**: Script-based. A bash script adds empty defaults (`requires:\n  language: []\nproject_affinity: []`) after the last frontmatter field before `---`. No per-template decisions needed.
- **Gold templates (78 files)**: Semi-automated. A mapping script uses existing `category` + `action` + `object` fields to infer reasonable defaults. For example: category=coding + action=review → `requires.language: []` (any language), `project_affinity: [web, api, library]`, `min_complexity: medium`. The mapping table is defined in the plan and reviewed before application. Manual override for templates with specific language needs (e.g., templates mentioning TypeScript → `requires.language: [typescript, javascript]`).
- **Rebuild index** after bulk update: `bash scripts/build-index.sh`

### F4: .cortex/project.json Configuration

Manual per-project config. Loaded by SessionStart and cortex-match.

```json
{
  "tech_stack": {
    "language": "typescript",
    "framework": "nextjs"
  },
  "boost": ["coding", "testing"],
  "suppress": ["legacy", "automation"],
  "disabled": ["coding-015"],
  "preset": "greenfield",
  "min_tier": "gold",
  "custom_templates": ".cortex/custom/"
}
```

**Behavior**:
- `tech_stack` overrides auto-detection (explicit > detected)
- `boost`/`suppress` accept category names OR template IDs
- `disabled` removes templates from candidate pool entirely
- `preset` activates a named preset (see F5)
- `min_tier` overrides the config.json tier setting for this project
- `custom_templates` path to team-shared templates directory

### F5: Presets System

Named configurations for different work modes.

**Built-in presets** (stored in `data/presets/`):

```json
// data/presets/greenfield.json
{
  "name": "greenfield",
  "description": "New project development — architecture, design, scaffolding",
  "boost": ["design", "create", "document"],
  "suppress": ["refactor", "fix", "debug"],
  "min_confidence_adjust": 0
}

// data/presets/maintenance.json
{
  "name": "maintenance",
  "description": "Bug fixes, performance, code quality",
  "boost": ["debug", "fix", "refactor", "optimize", "test"],
  "suppress": ["architecture", "design"],
  "min_confidence_adjust": 0
}

// data/presets/strict.json
{
  "name": "strict",
  "description": "Security-first, thorough testing, error handling",
  "boost": ["review", "test", "fix"],
  "suppress": [],
  "min_confidence_adjust": 0.05
}

// data/presets/learning.json
{
  "name": "learning",
  "description": "Explanations, documentation, lower threshold",
  "boost": ["explain", "document"],
  "suppress": [],
  "min_confidence_adjust": -0.10
}
```

**Preset matching**: boost/suppress values match against template `action` and `category` fields. All values MUST be valid actions (create, review, debug, refactor, explain, test, document, optimize, design, fix) or valid categories (coding, ai-workflows, research, automation, content, productivity). `min_confidence_adjust` shifts the global 0.70 injection threshold (capped at +/-0.10).

**Merge with project.json**: Preset rules are applied FIRST, then project.json rules overwrite. This matches the config precedence order (project.json rank 2, presets rank 4). In practice: preset provides a baseline, project.json fine-tunes.

**Activation**: `/cortex:preset <name>` writes to `.cortex/config.json`, or `"preset"` field in project.json.

### F6: /cortex:suggest — LLM-Powered CLAUDE.md Analysis

**The A+B hybrid approach**:

**Step A — Explicit command** (`/cortex:suggest`):
1. Read CLAUDE.md from project root (and subdirectory CLAUDE.md files)
2. Send to Claude with a structured analysis prompt
3. Extract: tech stack, testing philosophy, code style, quality standards, platform constraints
4. Generate proposed boost/suppress/disable rules with reasoning
5. Present to user for review
6. On acceptance, save to `.cortex/project-context.json`

**Step B — Cached at SessionStart**:
1. SessionStart checks for `.cortex/project-context.json`
2. If exists, load and merge with auto-detected context
3. No LLM call — pure file read
4. User runs `/cortex:suggest --refresh` when CLAUDE.md changes

**project-context.json schema**:
```json
{
  "generated_at": "2026-03-26T15:30:00Z",
  "claude_md_hash": "a7f3b2c1",
  "analysis": {
    "tech_stack": {
      "language": "typescript",
      "framework": "nextjs",
      "runtime": "bun"
    },
    "conventions": {
      "testing": "tdd with vitest",
      "style": "functional over class-based",
      "quality": "production-grade error handling",
      "constraints": ["no default exports", "biome for formatting"]
    }
  },
  "rules": {
    "boost": ["testing", "security", "review"],
    "suppress": ["legacy"],
    "disabled": []
  },
  "reasoning": {
    "boost_testing": "CLAUDE.md mentions TDD and vitest as testing standard",
    "suppress_legacy": "Modern stack (Next.js 14, Bun) — legacy patterns not relevant"
  }
}
```

**Flags**:
- `/cortex:suggest` — analyze and propose
- `/cortex:suggest --refresh` — re-analyze (CLAUDE.md changed)
- `/cortex:suggest --dry-run` — show proposals, don't save

**LLM boundary**: The LLM call happens in Claude's process via the markdown command file. No subprocess LLM calls. Layer 1 (the hook) never calls an LLM.

### F9: Complexity Scoring (Phase 2 Enhancement)

Add `min_complexity` to Phase 2 scoring:

```jq
# --- Complexity assessment ---
(prompt_word_count as $wc |
 if $wc < 6 then "trivial"
 elif $wc < 15 then "low"
 elif $wc < 40 then "medium"
 elif $wc < 80 then "high"
 else "expert" end) as $prompt_complexity |

# Complexity levels as numbers for comparison
({"trivial": 1, "low": 2, "medium": 3, "high": 4, "expert": 5}) as $levels |

(if ($levels[$prompt_complexity] // 3) < ($levels[$tmpl.min_complexity // "low"] // 2)
 then -0.15  # Already exists but only for <6 words. Extend to full range.
 elif ($levels[$prompt_complexity] // 3) > ($levels[$tmpl.min_complexity // "low"] // 2) + 2
 then -0.05  # Very complex prompt + simple template = mild mismatch
 else 0 end) as $complexity_penalty |
```

### F10: Domain Synonyms (intents.json v2)

Extend intents.json with context-sensitive synonyms:

```json
{
  "version": 2,
  "actions": [...],
  "objects": [...],
  "action_synonyms": { ... },
  "object_synonyms": { ... },
  "domain_synonyms": {
    "web": {
      "action_synonyms": {
        "create": ["scaffold", "bootstrap", "init"],
        "optimize": ["lighthouse", "lazy load", "code split"]
      },
      "object_synonyms": {
        "component": ["page", "layout", "route"],
        "API": ["endpoint", "handler", "middleware"]
      }
    },
    "data": {
      "action_synonyms": {
        "optimize": ["vectorize", "batch", "cache", "index"],
        "create": ["pipeline", "etl", "ingest"]
      },
      "object_synonyms": {
        "database": ["table", "collection", "index", "query"],
        "schema": ["model", "migration", "seed"]
      }
    },
    "devops": {
      "action_synonyms": {
        "create": ["provision", "deploy", "spin up"],
        "debug": ["triage", "incident", "on-call"]
      },
      "object_synonyms": {
        "config": ["dockerfile", "compose", "helm", "terraform"]
      }
    }
  }
}
```

**Language-to-domain mapping** (in intents.json v2):
```json
"domain_map": {
  "typescript": "web",
  "javascript": "web",
  "python": "data",
  "go": "devops",
  "rust": "devops",
  "java": "web"
}
```

Framework overrides language domain: `nextjs` → web, `fastapi` → web (even though python → data), `django` → web. The mapping table is: `"framework_domain_override": { "nextjs": "web", "react": "web", "vue": "web", "svelte": "web", "express": "web", "fastapi": "web", "django": "web", "flask": "web" }`.

**match.jq changes**: Determine active domain from `$context.framework` (via framework_domain_override) falling back to `$context.lang` (via domain_map). When a domain is active, merge that domain's synonyms with the base synonyms before scoring. If no domain matches, only base synonyms are used (current v1.1 behavior).

### F11: Template Packs

Domain-specific template bundles with auto-detection:

```
data/packs/
  react/
    pack.json
    react-001.md
    react-002.md
    ...
  python/
    pack.json
    ...

    pack.json

```

**pack.json schema**:
```json
{
  "name": "react",
  "description": "Prompt templates for React development",
  "version": "1.0.0",
  "signals_boost": ["react", "component", "hook", "useState", "jsx", "tsx"],
  "auto_detect": {
    "package_json_deps": ["react", "react-dom"],
    "file_patterns": ["**/*.tsx", "**/*.jsx"]
  },
  "auto_enable": true,
  "quality_tier": "silver"
}
```

**Behavior**: Pack templates are ALWAYS included in the index (build-index.sh includes all packs unconditionally). The context_filter in match.jq handles relevance at match time — pack templates have `requires.language` and `requires.framework` fields that Phase 3 uses to filter them for the current project. The `auto_detect` and `signals_boost` fields in pack.json are metadata for the context filter, not for build-time exclusion. This is consistent with the rest of the architecture: build once globally, filter per-project at match time.

**Scope for v1.2**: Ship pack infrastructure + 1-2 example packs (react, python). More packs in v1.3.

### F12.5: CLAUDE.md Staleness Detection (SessionStart enhancement)

At SessionStart, if `.cortex/project-context.json` exists:
1. Compute hash of CLAUDE.md (same hash algorithm as project-context.json's `claude_md_hash`)
2. Compare with stored `claude_md_hash`
3. If different, add a note to the SessionStart context message: `[cortex] CLAUDE.md has changed since last /cortex:suggest scan. Run /cortex:suggest --refresh to update project context.`
4. Do NOT re-scan automatically — the user chose the A+B hybrid model for a reason (user-approved rules only)

### F13: CwdChanged Hook

New hook script: `hooks/cortex-cwd-changed`

**Input contract** (stdin JSON from Claude Code):
```json
{
  "session_id": "abc123",
  "cwd": "/new/working/directory"
}
```

**Output contract** (stdout JSON):
```json
{
  "hookSpecificOutput": {
    "hookEventName": "CwdChanged",
    "additionalContext": "<cortex-context-update>Project re-detected: typescript/nextjs</cortex-context-update>"
  }
}
```

**Behavior**:
1. Read new cwd from stdin
2. Re-run tech stack detection (same logic as cortex-session-init's detect_project)
3. Re-load `.cortex/project.json` and `.cortex/project-context.json` from new cwd
4. Write updated context to CLAUDE_ENV_FILE (or `.cortex/session-context.json` fallback)
5. Output context update message

**Fallback** (if CwdChanged hook event doesn't exist): cortex-match compares `$cwd` against `lastCwd` in state file. If different, reads `.cortex/session-context.json` from new cwd and updates in-memory context. Single string comparison on every prompt — negligible cost.

Registered in hooks.json alongside existing hooks.

## Backwards Compatibility

All new features are additive and opt-in:

- **No project.json?** Phase 3 context filter returns input unchanged (current behavior)
- **No project-context.json?** No cached CLAUDE.md rules applied
- **No preset?** Default behavior, no boost/suppress
- **Templates without requires field?** Match against any project (current behavior)
- **Templates without project_affinity?** No affinity bonus (current behavior)
- **CwdChanged hook not supported?** SessionStart detection still works, just not reactive

Zero regression for existing users.

## Anti-Clippy Guarantees (preserved)

All v1.0/v1.1 anti-Clippy rules remain:
- Silence is default (confidence < 0.40 = nothing)
- Token budget (300 tokens max)
- Speed guarantee (< 2s, no LLM in Layer 1)
- Leave-it-alone detector (13 signals)
- Escape hatches (--raw, /cx off)
- Invisible Layer 1 (no "I enhanced your prompt!" messages)
- Never rewrites (inject alongside only)

**New v1.2 additions**:
- Context filter can only ADD filtering (removing bad matches), never force injection
- Boost/suppress adjustments are capped: boost max +0.10, suppress max -0.15
- Disabled templates are hard-removed — no override can bring them back
- Presets only adjust confidence threshold by max +/-0.10
